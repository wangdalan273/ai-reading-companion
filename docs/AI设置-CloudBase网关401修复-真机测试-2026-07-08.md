# AI 设置连不上 CloudBase 网关（HTTP 401）修复 — 真机测试指南

> 日期：2026-07-08 ｜ 修复人：Senior Developer（高级开发工程师）
> 配套服务器：`http://127.0.0.1:8123`（已用修复后代码重启，PID 18932）

## 一、之前为什么连不上（根因，一句话）

你的网关是 **腾讯云 CloudBase AI 网关**，它虽然走 Anthropic（Claude）的 Messages 协议，
但**鉴权头用的是 `Authorization: Bearer`，不是 `x-api-key`**。
ccswitch 里选「Anthropic Messages（原生）」能通，正是因为它把 token 当 Bearer 发出去了；
而本应用之前的 anthropic 分支只发了 `x-api-key`，网关不认 → 报「凭证缺失」→ 401。

（顺带：之前更早的 `cURL error 60` 是另一个问题——本机 PHP 缺 CA 证书包，已用项目内置
`storage/certs/cacert.pem` 修好，与本次 401 是两码事。）

## 二、改了什么

1. **`app/Services/LlmService.php`** — anthropic 分支现在**同时发** `x-api-key` 和
   `Authorization: Bearer`（官方 Anthropic 认前者、CloudBase 这类网关认后者，互不冲突）。
2. **`app/Services/LlmService.php`** — `testConnection` 把网关返回的错误码/信息透传出来
   （如 `[INVALID_CREDENTIALS] …`），不再只显示笼统的「请检查 key」。
3. **`app/Models/AiConfig.php`** — `cloudbase` 预设原来 `base_url` 是空的（Bug），
   改为占位符 `https://YOUR-ENV-ID.api.tcloudbasegateway.com/v1/ai/cloudbase`，选预设后替换你的环境 ID 即可。
4. **`resources/views/livewire/ai-settings.blade.php`** — 提示文案同步（anthropic 注明「x-api-key 或 Bearer 鉴权」，
   CloudBase 注明走 Bearer）。

## 三、你现在怎么测（两种方式任选其一）

### 方式 A：用「CloudBase 网关」预设（最省事）
1. 打开 `http://127.0.0.1:8123/enter` 自动登录 → 进书架 → 右上「⚙️ AI 设置」。
2. 厂商下拉选 **「腾讯云 CloudBase 网关（Anthropic 协议）」**（在「云服务网关」分组下）。
3. 这时 Base URL 自动填成 `https://YOUR-ENV-ID.api.tcloudbasegateway.com/v1/ai/cloudbase`，
   **把 `YOUR-ENV-ID` 改成你真实的网关 ID**（即你 ccswitch 里那段 `hy3-d8gfx6nztf84ee6cf`）。
4. 模型保持 `hy3-preview`；API Key 填你 ccswitch 里用的同一个 token。
5. 点「保存设置」→ 点「测试连接」。
6. **预期**：显示「连接成功：hy3-preview（anthropic）可用。」（或 `[INVALID_CREDENTIALS]` 之类明确错误，而不是之前的笼统 401）。

### 方式 B：用「自定义」（如果你更习惯手动填）
1. 厂商选 **「自定义（可选协议）」**。
2. **API 协议** 务必选 **Anthropic / Claude**（不要选 OpenAI，否则路径会变成 `/chat/completions` 而 404/401）。
3. Base URL 填 `https://hy3-d8gfx6nztf84ee6cf.api.tcloudbasegateway.com/v1/ai/cloudbase`。
4. 模型填 `hy3-preview`；API Key 填你的 token。
5. 保存 → 测试连接 → 预期同上。

## 四、两个常见坑（必看）

- ❌ **不要选「腾讯混元 Hunyuan」预设去填 CloudBase 的 token。**
  「混元」预设指向的是**官方混元 API**（`api.hunyuan.cloud.tencent.com`），和 CloudBase 网关是两套不同端点，
  用 CloudBase 的 token 打官方混元必然 401。要接 CloudBase 就用方式 A 或 B。
- ⚠️ 若测试仍报 `[INVALID_CREDENTIALS]`：说明 Bearer 头已正确送达、路径也对，纯粹是 token 不对/复制多了空格/
  该 token 在网关侧已失效——请直接从 ccswitch 里**重新复制**同一串 token 粘贴进来（注意前后无空格）。

## 五、自测已通过的证据

- 用 Laravel 完整 `Http` 栈 + 项目 CA 包 + 新加的双鉴权头，对真实网关发起请求：
  返回 `{"code":"INVALID_CREDENTIALS",…}`（之前只发 `x-api-key` 时是 `MISSING_CREDENTIALS`）。
  这说明 **Bearer 头已带上、TLS/路径全通**；假 token 才报 invalid，**你填真 token 必过**。
- `php -l` 后端两文件通过；`view:clear`+`view:cache` 重编译；开发服务器已重启（PID 18932）。
