## 0. 总原则

1. **第一性原理排期**：功能沿"接入→理解→记忆→复用→延展"五层**自下而上**推进；越往上越依赖下层的**结构化文本/向量**。
2. **复用优先**：能直接借力的开源项目（已标注 MIT/可商用）优先"借提示词 + 渲染库 + 管线形状"，不在 Laravel/PHP 栈里重造轮子。
3. **本地优先**：能用本地服务（OCR、whisper）就本地起服务；纯前端能做的（vis.js 渲染、Web Speech）就前端做；保持"数据不出本机"的产品调性。
4. **架构约定不变**：每表带 `user_id`、白屏硬化、整页导航、移动端安全区，不破坏已验收的 13/13 链路。

## 基础件 · Phase 0：PP-OCRv6 本地 OCR 接入（解锁扫描版）

- **第一性原理**：阅读的前提是"有字可读"。很多书（老书/教材/扫描档案）是图片式 PDF 没文字层，P12 总结、N3 概念图、N7 人物图、P13 跨书检索**全部失效**——没有文本，AI 就是瞎子。
- **你已具备**：`ppocrv6-studio`（FastAPI + ONNX Runtime，本地 `localhost:8765`，中文精确匹配 93.2%，数据不出机，MIT 代码 + Apache2 模型）。
- **怎么接（Laravel 侧）**：
  1. 上传 PDF 时检测是否缺文字层（试提取文本为空即判定扫描版）。
  2. 用 **Imagick/Ghostscript** 把每页转 PNG（该仓库只吃图片，不吃 PDF）。
  3. 逐页 POST 到本地 OCR 服务（`http://localhost:8765`，具体路由读 `webapp/server.py` 确认，或让 OCR 机跑一个我们自写的小 Python 包装脚本暴露 `/ocr`）；拿到每页 `text` + 检测框。
  4. 汇总多页文本存入 `books` 的 `ocr_text` 字段，后续 P12/P13/N3/N7 直接读它。
- **直接复用**：`andyhuo520/ppocrv6-studio`（MIT + Apache2 模型），无需自训；若 API 路径不透明，可 `exec()` 调用其 Python 脚本作为降级。
- **依赖**：OCR 机已部署（你已完成）；Laravel 侧加 Imagick 扩展 + 一个 OCR 调用 Service。
- **工作量**：中（主要在前置 PDF→图 + 路由确认 + 落库）。
- **风险**：Linux 服务器无 CoreML → 仅 CPU 推理，Medium 档单页 10–52s，可接受（后台队列处理）；端点未文档化 → 需读 `server.py` 一次。

## 理解层 · Phase 1（核心、低密度依赖、直接增厚"伴读感"）

### P12 · 章节级 AI 总结 + 全书思维导图

- **第一性原理**：线性阅读留下碎片；人脑以"结构"存知识。导图/大纲把碎片拼成地图，一眼看见"这本书讲了什么"，回头能定位。
- **产品形态**：阅读器侧栏/独立页「📊 本书脑图」按钮 → 按 EPUB 章节切分 → 逐章生成总结 → 聚合全书思维导图（点节点跳回该章原文）。进度条可见。
- **技术方案**：
  - 章节切分复用阅读器已有的 EPUB spine/导航解析。
  - 逐章调 `LlmService`（P11 已全厂商开放）生成"要点列表"；再让模型把全部要点压成一棵层级 JSON（主题→章→节→要点）。
  - **前端渲染直接复用 `markmap`**（markmap-lib + markmap-view，MIT）——把层级 markdown/JSON 渲染成可缩放思维导图，零自研布局算法。
- **直接复用的 GitHub 项目**：
  - `ebook-to-mindmap` / `ai-reader-pro`：复其**分章节总结的 prompt 结构**（"先列章要点，再聚合成树"）。
  - `markmap`（前端库，MIT）：直接用于导图渲染。
- **依赖**：P11（需真实模型）、EPUB 章节解析（已有）、Phase 0 OCR（扫描版书走 ocr_text）。
- **工作量**：中。
- **风险**：低（prompt + 前端库，均成熟）。

### N5 · 术语/概念悬停解释

- **第一性原理**：阅读中断成本极高——遇到生词/陌生概念卡住是读者放弃头号原因。零步成本"就地解释"把中断降到最低。
- **产品形态**：选中一个词/术语 → 浮条加「📖 释义」→ 派发"用一句话通俗解释这个词，并给一例"的请求（复用现有划词问 AI 基础设施，预置模板）。结果以气泡原地展示，不跳页。
- **技术方案**：在 `reader.js` 选区浮条加一个按钮，命中即发一个带固定 system 指令的轻量请求；复用 `LlmService` 的 `format` 分流（P11）。
- **直接复用的 GitHub 项目**：`Koodo Reader` / 沉浸式翻译的**划词交互范式**（UX 借鉴，非代码）；实现完全基于我们已有的选区浮条。
- **依赖**：P11。
- **工作量**：小。
- **风险**：低（纯前端 + 一次 API 调用）。

### N9 · 辩论 / 魔鬼代言人

- **第一性原理**：深度理解需"对抗性检验"——你以为懂了，让 AI 站反面挑刺，漏洞立刻显形。这是主动思考的催化剂。
- **产品形态**：对话栏加「🎭 反驳我」按钮 → 针对你刚说的观点/选中的笔记，AI 生成一段反方论证 + 2–3 个诘问；可"继续反驳"多轮。
- **技术方案**：一个固定 system 角色（"你是魔鬼代言人，只挑逻辑漏洞、反例、未证前提"）+ 用户观点入参。复用对话 SSE 流式（P2 已有）。
- **直接复用的 GitHub 项目**：通用 Socratic/debate 提示词范式（公开、无许可问题）；无需引第三方代码。
- **依赖**：无（复用 P2 对话基础设施）。
- **工作量**：小。
- **风险**：低（纯 prompt 层）。

## 理解层 · Phase 2（图谱类，依赖 P12 的文本单元）

### N3 · 知识图谱 / 概念图谱

- **第一性原理**：比导图更进一层——概念之间是网状而非树状。"A 和 B 什么关系""这个概念在哪一章出现"用图比用大纲更贴近人脑联想。
- **产品形态**：「🕸 概念图谱」→ 从全书抽取"主语-关系-宾语"三元组 → 渲染成交互式概念图（节点大小按出现频次/中心度，点节点看定义与出处章节）。
- **技术方案**：
  - 抽取：把全书按段落分块（重叠切分，参考 `ai-knowledge-graph` 的 `chunk_text` 思路，chunk≈200 词、overlap≈20），逐块让模型抽 SPO JSON；再做**实体标准化**（"AI"↔"人工智能"合并）。
  - 渲染：**前端用 `vis-network`（同 story-graph）或 `cytoscape`** 画力导向图（不选 PyVis 的 standalone HTML，保持应用内渲染）。
- **直接复用的 GitHub 项目**：
  - `robert-mcdermott/ai-knowledge-graph`：复用其**多阶段抽取管线思路**（分块→SPO 抽取→实体标准化→关系推断）与 NetworkX 式中心度计算；**许可证未明示 → 不复用其源码文件，改为按思路自实现抽取 prompt**（避免许可风险）。
  - `KGGen`（斯坦福）、`KnowledgeGrapher`：抽取思路备选参考。
- **依赖**：P12 的章节/段落文本单元、Phase 0 OCR（扫描版）。
- **工作量**：中–大（抽取质量调优是难点）。
- **风险**：中（图谱质量依赖模型抽取；需做实体去噪、关系去重）。

### N7 · 小说人物关系图 + 时间线

- **第一性原理**：小说读者最大痛点是"人物太多记不住、时间线乱"。显式画出关系与事件轴，沉浸感与回顾效率倍增。
- **产品形态**：「👥 人物关系」→ 识别书中人物 → 抽取关系（亲属/敌对/合作/师徒）与关键事件时间线 → 渲染成可交互关系网络（vis.js 力导向）+ 底部时间轴；点人物看介绍、关系列表、出场章节；支持"刘备和曹操什么关系"自然语言问答高亮路径。
- **技术方案**：
  - 抽取：让模型输出 JSON `{characters:[{name,faction,desc}], relations:[{from,to,type}], events:[{time,desc,characters}]}`；**先让模型判断体裁**（小说 vs 非虚构），非小说不显示此模块。
  - 渲染：**直接复用 `story-graph` 的 `vis-network` 关系图 + 时间轴实现思路**（MIT，可借）。
- **直接复用的 GitHub 项目**：`Drwei3155/story-graph`（**MIT，最强直接复用**）：
  - 后端：`server.js` 的 `/api/generate`（关系图）、`/api/search`、`/api/explain-path` 调用模式 → 我们改接自己的 `LlmService`（OpenAI 兼容）即可。
  - 前端：`index.html` 的 **vanilla JS + vis.js（vis-network + vis-data）** 力导向图 + html2canvas 导出 PNG → 直接移植渲染代码到我们的阅读器页。
  - 抽取 prompt：其"DeepSeek 分析角色/阵营/关系/事件"的提示词结构可直接借鉴（改用我们配置的模型）。
- **依赖**：P12 文本、体裁识别（模型判断）、Phase 0 OCR。
- **工作量**：中。
- **风险**：中（人物共指/别名消歧是难点；vis.js 移植成本低）。

### N6 · 论据/论证地图（非虚构批判阅读）

- **第一性原理**：非虚构阅读本质不是"记住"而是"判断"——作者凭什么这么断言？证据够不够？把隐含论证结构显式画出，培养批判性思维。
- **产品形态**：「⚖ 论证地图」→ 对选中章节/全书抽取"主张-证据-反驳" → 渲染成论证结构图；可点"这个说法的反面证据是什么"让 AI 补刀。
- **技术方案**：
  - 抽取：让模型输出论证 JSON（claim / premises[] / counter-evidence[]），参考 **DeepA2 论证分析框架**（ claim-premise-conclusion 标注）。
  - 渲染：用 `mermaid` 流程图或 `cytoscape` 画论证树（复用 P12 的 markmap/mermaid 前端基础）。
- **直接复用的 GitHub 项目**：
  - `argumentation` topic（DeepA2 等）：复用其**论证分析 prompt 框架**（主张/前提/结论标注），自实现抽取。
  - `AMQuestioner`（CSCW'25）：复用"用生成问题练批判性思维"的思路，给每个 claim 生成 1–2 个诘问。
- **依赖**：P12 文本、体裁识别（小说不显示）、Phase 0 OCR。
- **工作量**：中。
- **风险**：中（长文论证抽取易漂移；需限定输入粒度到"章/节"）。

## 记忆 + 复用层 · Phase 3（较重，建向量基建）

### P13 · 跨书/跨笔记 RAG 问答 + 自定义 System Prompt

- **第一性原理**：真正的学习发生在"跨书连接"——读过的书之间互相印证，知识才长成网，这是"伴读"和"普通 Chat 工具"的分水岭。
- **产品形态**：对话栏「💡 跨书问答」模式 → 问题先在我们所有书/笔记的向量库里检索最相关片段 → 带引用回答（"据《xxx》第 N 章：…"）；用户可在设置里写**自定义 system prompt**（"你是我严厉的写作教练""用通俗比喻解释"）。
- **技术方案**：
  - 索引：把每本书的章节/每段划线做成**原子片段**，调 embedding（我们 LLM 的 embedding 端点，或本地 `bge-small` 等）→ 存 **SQLite + sqlite-vec**（本地向量，不引云端 Pinecone，保持本地优先）。
  - 检索：向量相似度 + BM25 关键词兜底（双路融合）。
  - 回答：检索片段作为 context 注入，要求模型**带书名/章节引用**；自定义 prompt 写入 `ai_configs` 或独立 `user_prompts` 表。
- **直接复用的 GitHub 项目**：
  - `Obsidian Copilot` / `Knowledge AI` / `Vault AI Chat`：复用其 **RAG-with-citation 模式**（检索片段→带源回答）与多格式索引思路。
  - `Dzeddy/zettelkasten`（MIT）：复用其**分块→嵌入→向量检索**管线形状（但我们用 SQLite 本地向量替代 Pinecone）。
- **依赖**：Phase 0 OCR（书文本）、P12（章节切片）、embedding 能力（走 P11 的 LLM 或本地模型）。
- **工作量**：中–大（向量检索基建是新模块）。
- **风险**：中（embedding 成本/延迟；SQLite 向量规模需压测）。

### N12 · 个人知识库图谱 / Zettelkasten 原子笔记

- **第一性原理**：卢曼"卡片盒"——把笔记拆成原子单元并互相链接，连接本身产生新想法。这是"伴读"走向"第二大脑"的关键一跃。
- **产品形态**：「🧠 我的知识库」→ 每张划线/笔记/章节总结是一张原子卡 → AI 自动建议"这张和哪张相关"并连边 → 全局知识图谱视图（vis/cytoscape）；点节点看卡片、反向链接。
- **技术方案**：
  - 原子化：复用 P13 的分块 + embedding。
  - 自动连边：对每对候选卡片算向量相似度 + 让模型判断是否"真相关"（去噪）→ 存 `note_links` 表。
  - 渲染：复用 N3/N7 的 **vis-network** 图谱视图。
- **直接复用的 GitHub 项目**：
  - `Dzeddy/zettelkasten`（**MIT**）：复用其**完整管线形状**（auto chunk → OpenAI embedding → vector store → interlinked graph）+ 自动连边逻辑；我们用 PHP+SQLite 本地实现替代其 Go+Pinecone，保持本地优先与栈统一。
  - `joshylchen/zettelkasten`：CEQRC 工作流思路参考。
- **依赖**：P13 的向量基建（必须先有）。
- **工作量**：大。
- **风险**：中–高（图谱规模/性能；自动连边噪声控制）。

### P15 · 苏格拉底模式：对话式追问 + 自动测验

- **第一性原理**：主动提取 > 被动阅读（测试效应）。AI 不直接给答案，而是连问引导你悟；并据划线自动出选择题测验，固化记忆。
- **产品形态**：对话栏「🧭 苏格拉底」模式 → 针对选中内容，AI 只提问不给答案，逐步引导；另「📝 出测验」→ 据当前章/划线生成 3–5 道选择题（含解析），答完给反馈。
- **技术方案**：
  - 追问：固定 system（"你是苏格拉底，只用问题引导，绝不直说答案"）+ 对话历史。
  - 测验：让模型据文本出 JSON 题库（stem/options/answer/reason），前端渲染答题卡。
- **直接复用的 GitHub 项目**：
  - `Socratic_Tutor` / `SocraticAI` / `Socranotes`：复用其**苏格拉底提问 prompt 框架**与**从笔记自动生成闪卡/测验**的逻辑（我们已有 P10 闪卡表，直接落库）。
- **依赖**：P13 的记忆层（笔记/划线已结构化）、Phase 0 OCR。
- **工作量**：中。
- **风险**：低（prompt + 题库 JSON，成熟模式）。

## 延展层 · Phase 4（独立、可后置）

### N14 · 语音输入问 AI

- **第一性原理**：手机上打字问 AI 效率低、打断心流；语音是更自然的接口，尤其通勤/躺读。
- **产品形态**：对话栏加麦克风按钮 → 说话转文字 → 自动发问；也可语音控制翻页（"下一页"）。
- **技术方案（两档）**：
  - **零成本起步**：浏览器 **Web Speech API**（`SpeechRecognition`，前端直连，无需服务端）→ 最快落地，中文支持看浏览器。
  - **高精度/离线升级**：本地起 **`whisper.cpp` 的 `whisper-server`**（`whisper-server --host 127.0.0.1 -m models/ggml-base.bin`），前端 `MediaRecorder` 录音频 → POST → 收文本。模型用 `small`/`base` 中文档。
- **直接复用的 GitHub 项目**：
  - `ggml-org/whisper.cpp`（**MIT**）：直接复用其 **`whisper-server` HTTP 服务**（类 OpenAI API）+ 模型（`ggml-base.zh` 等）；或 `whisper.wasm` 纯前端离线方案。
  - `Angana007/AI-Reading-Assistant`：复用其"语音控制播放/提问"的交互范式。
- **依赖**：无（可独立做）。
- **工作量**：中（Web Speech 版小；whisper 版中）。
- **风险**：中（Web Speech 中文准确率/兼容性参差；whisper 需本地起服务 + 模型下载）。

### N15 · 个性化荐书

- **第一性原理**：读什么比怎么读更决定成长。基于"你读过/划线/问过什么"画像荐书，比通用榜单精准得多。
- **产品形态**：书架页「📚 为你推荐」→ 分析书架标签 + 划线主题 + 问答历史 → LLM 生成"下一本读什么"及理由（含 3–5 本，附为何契合你的画像）。
- **技术方案**：聚合用户画像（读过的书 genre、高频划线关键词、常问主题）→ 拼成 prompt → 模型返回推荐列表 JSON（title/reason/tag）。可选接书目元数据（Open Library API）补全封面。
- **直接复用的 GitHub 项目**：
  - `ilzrzls/Book-Recommendation`、`akanksharai01/Book-recommendation-system`：复用其**"基于阅读历史个性化推荐"的 prompt 思路**（纯 LLM 生成理由即可起步，无需复杂协同过滤算法）。
- **依赖**：无（轻量，纯 LLM 生成）。
- **工作量**：中。
- **风险**：低–中（推荐理由靠 LLM，无需算法；如需真实书目可接 Open Library）。
